# Data Model: MLflow Experiment Tracking

**Feature**: `003-mlflow-experiment-tracking`
**Date**: 2026-02-16

## Entities

### Experiment (MLflow-managed, not in backend database)

MLflow internally manages experiments. The backend proxy adds a user
prefix for isolation but does not store experiment metadata separately.

| Field | Type | Constraints | Source |
|-------|------|-------------|--------|
| experiment_id | String | PK, auto-generated by MLflow | MLflow |
| name | String | UNIQUE, user-prefixed (e.g., `scientist1/my-experiment`) | Backend proxy |
| artifact_location | String | S3 URI in MinIO bucket | MLflow |
| lifecycle_stage | String | `active` or `deleted` | MLflow |
| creation_time | Long | Unix timestamp milliseconds | MLflow |
| last_update_time | Long | Unix timestamp milliseconds | MLflow |

**Notes**:
- MLflow manages its own database schema in the `mlflow` PostgreSQL
  database. The backend does NOT create its own experiment tables.
- The backend proxy transparently prefixes experiment names with the
  authenticated user's username for isolation (FR-007).
- When listing experiments, the backend filters by prefix and strips
  it before returning to the frontend/notebook.

### Run (MLflow-managed)

| Field | Type | Description |
|-------|------|-------------|
| run_id | String (UUID) | Unique run identifier |
| experiment_id | String | FK → Experiment |
| status | String | `RUNNING`, `SCHEDULED`, `FINISHED`, `FAILED`, `KILLED` |
| start_time | Long | Unix timestamp milliseconds |
| end_time | Long | Unix timestamp milliseconds |
| artifact_uri | String | S3 path for this run's artifacts |
| user_id | String | Username who created the run |

**Associated data**:
- **Parameters**: Key-value pairs (e.g., `learning_rate: 0.01`)
- **Metrics**: Timestamped numerical values (e.g., `accuracy: 0.95`)
- **Tags**: Arbitrary key-value metadata
- **Artifacts**: Files stored in S3 (models, plots, data)

### Artifact (S3/MinIO storage)

| Attribute | Value |
|-----------|-------|
| Storage | MinIO bucket `ml-platform-mlflow` |
| Path structure | `artifacts/{experiment_id}/{run_id}/artifacts/` |
| Access | Proxied through MLflow `--serve-artifacts` |
| Max size | No hard limit for MVP (MinIO capacity-bound) |

**Notes**:
- Artifacts are stored in MinIO's S3-compatible storage.
- Notebooks upload artifacts through the MLflow client, which sends
  them to the tracking server, which proxies to MinIO.
- Direct MinIO access from notebooks is NOT required.

## Infrastructure Entities

### MinIO Bucket Configuration

| Attribute | Value |
|-----------|-------|
| Bucket name | `ml-platform-mlflow` |
| Service | MinIO on K8s (Bitnami Helm chart, single-node) |
| Access | Via Kubernetes Secret (AWS_ACCESS_KEY_ID/SECRET) |
| Endpoint | `http://minio.ml-platform.svc:9000` |

### MLflow Database (PostgreSQL)

| Attribute | Value |
|-----------|-------|
| Database name | `mlflow` |
| Host | `postgresql.ml-platform.svc:5432` |
| Schema | Managed by MLflow server (auto-created on first start) |
| Shared with | Keycloak (separate DB), backend `users`/`workspaces` tables (separate DB) |

**Notes**:
- MLflow creates its own tables (`experiments`, `runs`, `metrics`,
  `params`, `tags`, `registered_models`, etc.) on startup.
- No Flyway migration needed — MLflow handles its own schema.

## Relationships

```
User (feature 001)
  ↓ (authenticated via JWT)
Backend Proxy (experiment endpoints)
  ↓ (user-prefixed API calls)
MLflow Tracking Server
  ↓ (metadata)          ↓ (artifacts)
PostgreSQL (mlflow DB)  MinIO (ml-platform-mlflow bucket)
```

- Users → Experiments: One-to-many (via username prefix in experiment
  name)
- Experiments → Runs: One-to-many (MLflow internal relationship)
- Runs → Artifacts: One-to-many (S3 path hierarchy)
